## 一、背景与目标

### 1.1 背景

业务团队在规划 AI Pipeline（如 LLM 推理链路、多模型串联链路等）时，需要快速评估端到端成本。当前缺乏一个直观的工具来：

- 可视化地组合 Pipeline 各模块
- 自动计算不同流量下的总成本
- 识别容灾要求带来的额外成本
- 快速对比多套配置方案的成本差异
- 将评估结果分享给客户或协作方

### 1.2 目标

构建一个 **可视化、可配置、具备容灾感知能力的 Pipeline 成本估算服务**，具体包括：

| # | 目标 | 说明 |
|---|------|------|
| 1 | 可视化拖拉拽编排 | 用户通过拖拉拽自由组合 Pipeline 模块，支持 DAG（有向无环图）拓扑，包括分支与合并 |
| 2 | 灵活配置 | 每个模块可独立配置性能参数（QPS/并发）和单实例成本，支持从资源规格库直接选择算力单元 |
| 3 | 自动成本计算 | 基于用户输入的目标 QPS/并发，自动计算全链路成本，并给出单 QPS / 单并发的日成本；支持端到端附加系数 |
| 4 | 容灾感知 | 自动判断是否满足最小部署要求，透明展示容灾补齐带来的额外成本及可额外承载的流量 |
| 5 | 数据持久化 | 支持 Pipeline 配置的保存、加载、复制、删除、加载后编辑，支持版本历史与回溯 |
| 6 | 多用户支持 | 基于百度员工邮箱的登录体系，区分普通用户与管理员角色 |
| 7 | 模块模板库 | 支持将常用模块的性能和成本预制保存为模板，方便复用 |
| 8 | 资源规格库 | 从数据库读取算力资源规格及价格，配置模块时可直接搜索选择 |
| 9 | 成本对比 | 支持多套 Pipeline 配置横向对比 |
| 10 | 分享功能 | 支持将计算后的 Pipeline 图和成本结果生成分享链接，可设置有效期 |
| 11 | 报告导出 | 支持将计算结果导出为 PDF 报告 |
| 12 | 端到端附加系数 | GPU 服务搭配 CPU 服务、存储等附加开销，支持在最终单价上乘以系数（1.X）得到端到端成本 |
| 13 | Pipeline 复制 | 支持复制已有 Pipeline 为新副本，可在副本上编辑并改名另存 |
| 14 | 成本趋势分析 | 基于历史版本的成本变化曲线，支持手动选择版本和方案 |
| 15 | 流量动画 | Pipeline 编辑完成后，动态可视化展示流量从入口流入、经分流、到出口流出的过程 |

## 三、核心概念定义

### 3.1 Pipeline（流水线）

一个由多个 **Module**（模块）和 **Edge**（连线）组成的 **DAG（有向无环图）**。

- 有一个或多个**入口节点**（无入边的节点）
- 有一个或多个**出口节点**（无出边的节点）
- 支持**分支**（一个节点的输出连到多个下游）和**合并**（多个节点的输出连到同一个下游）

### 3.2 Module（模块）

Pipeline 中的一个处理单元。每个模块被拖入画布后为空白初始状态，用户需配置：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | string | 是 | 模块名称（用户自定义，如"LLM 推理"、"向量检索"） |
| resource_spec_id | UUID | 否 | 关联的资源规格（从资源规格库选择，选择后自动填充成本字段） |
| performance_mode | enum | 是 | 性能指标类型：`qps`（每秒请求数）或 `concurrency`（并发数） |
| performance_value | number | 是 | 单实例可支撑的性能值 |
| avg_response_time | number | 条件必填 | 单请求平均响应时间（秒），用于 QPS ↔ 并发互转；当需要转换时弹窗提示用户填入 |
| gpus_per_instance | number | 是 | 每实例 GPU 数量（默认 1），用于统一按卡计费 |
| cost_per_unit_per_day | number | 是 | 单卡日成本（元/天），可从资源规格库自动带入或手动填写 |
| gpus_per_machine | number | 否 | 单机 GPU 数量（用于容灾 2 机模式计算最小实例数，常见如 4、8 等） |

### 3.3 Edge（连线）

连接两个模块的有向边。当一个模块有多条出边（分支）时，用户需为每条边配置**分流比例**（所有出边的比例之和 = 100%）。

### 3.4 全局输入

用户在 Pipeline 层面需配置：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| target_mode | enum | 是 | 目标指标类型：`qps` 或 `concurrency` |
| target_value | number | 是 | 目标 QPS 或目标并发数 |
| e2e_avg_response_time | number | 条件必填 | 端到端平均响应时间（秒），用于 QPS ↔ 并发互转 |
| e2e_cost_coefficient | number | 否 | 端到端成本附加系数，默认 `1.0`；用于将纯 GPU 成本乘以系数得到包含 CPU、存储、网络等附加开销的端到端成本。例如填 `1.3` 表示附加 30% 开销 |

## 四、资源规格库

### 4.1 概述

系统维护一个**算力资源规格与价格数据库**，用户在配置模块成本时可直接搜索和选择算力单元，无需手动填写价格。

### 4.2 数据来源

资源规格数据由一个**独立的 Python 爬取脚本**维护（不包含在本产品服务中），定期更新到数据库。本系统仅负责**读取和展示**。

### 4.3 ResourceSpec（资源规格）数据结构

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| serial_no | INTEGER | 序号（用于排序展示） |
| spec_name | VARCHAR(255) | 资源规格名称，如 `NVIDIA-V100-16G`、`NVIDIA-A100-40G` |
| monthly_price | DECIMAL(10,2) | 本月折后价格（元/月） |
| daily_cost | DECIMAL(10,2) | 每日成本（元/天） |
| updated_at | TIMESTAMP | 数据最后更新时间 |

### 4.4 数据库表示例

| 序号 | 资源规格 | 本月折后价格 | 每日成本 |
|------|----------|------------|---------|
| 1 | NVIDIA-V100-16G | ¥206.62 | ¥41.32 |
| 2 | NVIDIA-V100-32G | ¥229.58 | ¥45.92 |
| 3 | NVIDIA-A100-40G | ¥480.00 | ¥96.00 |
| 4 | NVIDIA-A100-80G | ¥680.00 | ¥136.00 |

## 五、模块模板库

### 5.1 概述

用户可将配置好的模块**保存为模板**，后续创建新 Pipeline 时直接从模板库拖入，免去重复配置。

### 5.2 模板类型

| 类型 | 创建者 | 可见范围 |
|------|--------|---------|
| **个人模板** | 普通用户 | 仅创建者可见 |
| **全局模板** | 管理员 | 所有用户可见（如预置的"LLM 推理"、"向量检索"等常用模块） |

### 5.3 ModuleTemplate（模块模板）数据结构

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| name | VARCHAR(255) | 模板名称，如"LLM 推理 - A100" |
| user_id | UUID | 创建者（NULL 表示全局模板） |
| is_global | BOOLEAN | 是否为全局模板 |
| config | JSONB | 模块配置快照（性能参数、成本参数、关联的资源规格等） |
| created_at | TIMESTAMP | 创建时间 |
| updated_at | TIMESTAMP | 更新时间 |

### 5.4 模板操作

| 操作 | 说明 |
|------|------|
| 保存为模板 | 在模块配置面板中点击"保存为模板"，输入模板名称即可 |
| 从模板创建 | 左侧模板库拖入画布即自动填充配置 |
| 编辑模板 | 在左侧"我的模板"分区中，点击模板卡片在右侧抽屉编辑并保存；也可通过编辑图标重命名 |
| 删除模板 | 在左侧"我的模板"分区中，点击删除图标可删除模板（需确认）；全局模板仅管理员可删除 |

### 6.2 模块面板（左侧）

- **统一模板库**：左侧面板为统一的"模板库"视图

  - "全局模板"区：管理员预置的模板，拖入画布自动填充配置

### 6.3 画布区域（中央）

- 基于 DAG 编辑器实现
- 支持操作：
  - **拖入模块**：从左侧模板库拖入（空白模块或模板）

  - **流量动画**：计算完成后，画布上动态展示流量流动效果（详见 6.9）

### 6.7 QPS ↔ 并发换算规则

| 场景 | 公式 | 所需输入 |
|------|------|----------|
| QPS → 并发 | `并发 = QPS × 平均响应时间` | 平均响应时间 |
| 并发 → QPS | `QPS = 并发 / 平均响应时间` | 平均响应时间 |

**交互规则**：
- 当用户切换指标类型时，若未填写平均响应时间，弹窗提示："请输入平均响应时间（秒）以完成换算"
- 换算结果实时显示在输入栏旁，不替换原始输入值

### 6.9 流量动画（Flow Animation）

计算完成后，画布上以**动态粒子动画**的方式展示流量在 Pipeline 中的流动过程，帮助用户直观理解流量分配。

#### 动画规则

| 元素 | 效果 |
|------|------|
| **连线上的粒子** | 沿连线方向持续流动的小圆点（或光点），代表请求流量 |
| **粒子速度** | 与该边的流量大小正相关（流量越大，粒子越密/越快） |
| **粒子密度** | 与该边的 QPS 成正比（QPS 高的边上粒子更密） |
| **分支处** | 粒子按分流比例分散到各出边，比例数字标注在分叉点旁 |
| **合并处** | 多路粒子汇聚到下游节点 |
| **边上标注** | 每条边上方显示该边的 QPS 数值和分流比例（如 `700 QPS / 70%`） |
| **节点标注** | 节点卡片内显示该节点接收的总 QPS |

#### 交互

```
┌───┐  ─── 700 QPS (70%) ──▶  ┌───┐  ─── 1000 QPS ──▶  ┌───┐
│ A │  ● ● ● ● ● ● ● ● ● ●   │ B │  ● ● ● ● ● ● ●     │ D │
│1000│                          │700│                      │1000│
│QPS │  ─── 300 QPS (30%) ──▶  └───┘                      └───┘

## 七、成本计算引擎

### 7.1 流量分配（DAG 遍历）

采用**拓扑排序 + 正向传播**计算每个节点的实际 QPS：

```
1. 对 DAG 进行拓扑排序
2. 入口节点的 QPS = 用户输入的目标 QPS（如输入为并发，先转换为 QPS）
3. 按拓扑序遍历每个节点：
   a. 节点的入流量 = 所有入边流量之和
   b. 节点的出流量按分流比例分配到各出边
4. 合并节点（多条入边）：流量 = Σ(各入边流量)
```

### 7.2 单节点实例数计算

```
node_qps          = 该节点通过 DAG 分配到的 QPS
qps_per_instance  = 单实例性能值（如模块配置为并发，则先转换为 QPS）

raw_instances     = ceil(node_qps / qps_per_instance)
```

### 7.3 单节点成本计算（统一按卡计费）

```
total_gpus = final_instances × gpus_per_instance
node_cost  = total_gpus × cost_per_unit_per_day
```

> **说明**：已取消 `per_gpu` / `per_machine` 区分，统一按卡计费。`gpus_per_instance` 表示每个实例占用的 GPU 数量（默认为 1）。`gpus_per_machine` 仅用于容灾 2 机模式下计算最小实例数。

### 7.4 全链路汇总

```
GPU 成本汇总（不含附加系数）：
gpu_total_cost_per_day     = Σ(所有节点的 cost)

端到端成本（含附加系数）：
e2e_total_cost_per_day     = gpu_total_cost_per_day × e2e_cost_coefficient

单位成本（均基于端到端成本）：
e2e_cost_per_qps_per_day   = e2e_total_cost_per_day / target_qps
e2e_cost_per_conc_per_day  = e2e_total_cost_per_day / target_concurrency
```

> **说明**：每个 GPU 服务都搭配一定量的 CPU 服务、存储和 CPU 架构串/并联服务。端到端附加系数（默认 1.0）用于覆盖这些额外开销。例如系数 1.3 表示在纯 GPU 成本基础上附加 30% 的 CPU/存储成本。

## 八、容灾（HA）能力

### 8.1 容灾入口

计算结果展示后，提供"容灾评估"开关。用户启用后进入容灾配置流程。

### 8.2 容灾配置交互

```
┌──────────────────────────────────────────────┐
│  容灾配置                                 ✕  │
├──────────────────────────────────────────────┤
│                                              │
│  最小容灾单元：                              │
│  (○) 2卡（至少 2 张 GPU 卡）                 │
│  (●) 2机（至少 2 台物理机）                  │
│                                              │
│  (若选2机) 每台机器GPU数量：[    ]           │
│                                              │
│         [ 确认并计算 ]  [ 取消 ]             │
└──────────────────────────────────────────────┘
```

**说明**："每台机器 GPU 数量"为用户自行填写，支持任意正整数（如 4、8、16 等），不做固定预设。

### 8.3 容灾计算逻辑

**对每个节点独立判断：**

```
Step 1：确定最小实例数
  若容灾单元 = 2卡：
      min_instances = 2

  若容灾单元 = 2机，每机 M 卡（用户填入）：
      min_instances = 2 × M

Step 2：容灾补齐
  final_instances = max(raw_instances, min_instances)

Step 3：标记
  ha_adjusted        = (final_instances > raw_instances)
  ha_extra_instances = final_instances - raw_instances
```

### 8.4 容灾后可承载流量计算

**关键能力**：用户关心"多花的钱还能承接多少流量"。

```
ha_supported_qps         = final_instances × qps_per_instance
ha_extra_qps_capacity    = ha_supported_qps - node_qps

全链路维度（取瓶颈）：
ha_total_extra_capacity  = min(各节点按 DAG 反推到入口的额外 QPS 容量)
```

### 8.5 容灾成本汇总

```
单节点：
  ha_extra_cost = (final_instances - raw_instances) × cost_per_unit_per_day
  （若 cost_unit = per_machine，按机器粒度计算）

全链路（GPU 维度）：
  ha_impact_cost = Σ(所有节点的 ha_extra_cost)

全链路（端到端，含附加系数）：
  ha_e2e_impact_cost       = ha_impact_cost × e2e_cost_coefficient
  ha_e2e_total_cost_per_day = e2e_total_cost_per_day + ha_e2e_impact_cost
```

### 8.6 容灾结果展示增强

容灾计算后，除原有成本信息外，还需明确展示：

```

## 十九、验收标准

### 19.1 功能验收

| # | 验收项 | 标准 |
|---|--------|------|
| 1 | 用户注册/登录 | 百度邮箱格式校验，密码 >= 6 位，JWT 鉴权正常 |
| 2 | 角色权限 | 管理员可管理全局模板、查看所有 Pipeline、管理用户角色 |
| 3 | Pipeline 拖拉拽 | 模块可从模板库拖入画布（空白模块和模板均可），模块间可连线，支持删除 |
| 4 | DAG 支持 | 支持分支和合并拓扑，分流比例可配置且校验合计 = 100% |
| 5 | 模块配置 | 所有字段可正确配置，QPS/并发可互相换算，可从资源规格库选择算力单元 |
| 6 | 资源规格库 | 可搜索、选择资源规格，选择后自动填充日成本 |
| 7 | 成本计算正确性 | DAG 流量分配正确，实例数计算正确，成本汇总正确，端到端附加系数正确应用于所有成本指标（含容灾） |
| 8 | 容灾补齐 | 自动判断、补齐实例数，GPU 数量用户可自由配置（非固定值），标记哪些节点被调整 |
| 9 | 容灾额外容量 | 正确计算补齐后的机器数、GPU 数、可额外承载的 QPS/并发 |
| 10 | 成本透明 | GPU 成本、端到端成本（含系数）、单位成本、容灾额外成本均清晰展示 |
| 11 | 数据持久化 | Pipeline 可保存、加载、编辑后再保存、另存为、复制、删除 |
| 12 | 版本历史 | 每次保存自动生成版本（配置无变化时跳过），可查看历史，可回溯到任意版本 |
| 13 | 模块模板 | 可保存模块为模板，可从模板创建模块，全局模板所有人可见 |
| 14 | 成本对比 | 可选择多个 Pipeline/版本进行横向对比，自动标注最优方案 |
| 15 | 分享功能 | 可生成带有效期的分享链接，接收方无需登录即可查看只读结果 |
| 16 | PDF 导出 | 可导出包含拓扑图和成本明细的 PDF 报告 |
| 17 | 多用户隔离 | 普通用户只能看到自己的 Pipeline 和个人模板 |
| 18 | 端到端附加系数 | 用户可设置系数（默认1.0），所有成本输出均乘以该系数，容灾成本同样适用 |
| 19 | Pipeline 复制 | 可复制已有 Pipeline 为新副本，副本可独立编辑和改名 |
| 20 | 成本趋势分析 | 可选择多个版本/方案，查看成本变化趋势曲线，支持切换指标维度 |
| 21 | 流量动画 | 计算完成后画布上动态展示流量流动粒子效果，边上标注 QPS 和分流比例 |
